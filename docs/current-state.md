## What this repo is

The **websites** repo for the menno420 estate — one repo, multiple Railway
services (share code, not a process):
- **control-plane** (`app/`) — **public** readiness board + journal browser over
  `superbot`, `superbot-next`, `substrate-kit`, and `websites` itself, live from
  the GitHub API. The one non-public datum (Actions secret names) is masked to a
  count on the public board; a password-gated `/owner` area ([D-0012], also
  reachable via the fleet Discord login) un-masks it and exposes privileged
  actions (cache refresh, re-run failed CI). Full doc: `docs/site.md`.
- **botsite** (`botsite/`) — the public marketing + reference site; also hosts
  the `/submit` idea intake and the `/testing` tester program, both now
  **Postgres-backed** (durable, dual-backend SQLite⇄Postgres). Doc:
  `docs/botsite.md`.
- **dashboard** (`dashboard/`) — the **public** developer dashboard: read-only
  oversight of the bot's own inventory (functions/commands/settings/access/env/
  ideas/bugs). The live-bot-write `/admin` panel is now gated behind the fleet
  Discord login (dry-run actions only; the read views stay public). Doc:
  `docs/dashboard.md`.
- **review** (`review/`) — the program-review site for Anthropic reviewers
  (owner-directed 2026-07-11): process / growth / successes / problems / fleet
  index + per-repo detail / continuous review editions (+ Atom feed) /
  evidence-backed questionnaire + questions ledger, rendered server-side from
  committed data mirrors (`review/data/*.json` + `review/data/reviews/*.md`,
  baked by `review/gen_*.py`, refreshed by the scheduled `review-bake`
  workflow). Read-only, network-free at runtime. LIVE at
  https://review-production-fc91.up.railway.app (owner-created 2026-07-12).

**Bake pipeline (hands-off since 2026-07-18):** the scheduled `review-bake`
workflow regenerates the snapshot/fleet/stats mirrors and lands the data-only
diff itself. `BAKE_PAT` (a fine-grained PAT, wired #434/#438) authors the bake
PR under a non-`GITHUB_TOKEN` identity, so it now gets real `pull_request`
quality checks and auto-merges — the old GitHub recursion-guard wall (no checks
under the ambient `GITHUB_TOKEN`; ruleset `405 Repository rule violations
found` on the dispatch substitute) is closed. That wall's history:
`docs/CAPABILITIES.md`.

All FOUR services are **public** — the Basic-auth gate was dropped from
control-plane + dashboard ([D-0011]); botsite and review were always public.
The gated corners are the control-plane `/owner` area and dashboard `/admin`,
both via the fleet Discord login (`SITE_PASSWORD` remains an optional
control-plane/botsite fallback).

Deployment: `docs/deployment.md` + each service's doc.

## Review rhythm

Every session: born-red `.sessions/<date>-<slug>.md` log first commit → work →
flip `complete` last; decisions appended to `docs/decisions.md` as [D-NNNN];
owner questions to `docs/question-router.md`;
`python3 bootstrap.py check --strict` green before PR; fresh branch → PR,
forward-only git. Keep this ledger current at every session close.

**Landing doctrine.** Open PRs **READY** with green CI, then merge your own
green PR directly (MCP/REST) — or let the server-side `auto-merge-enabler`
workflow land it on green. Merging your own green PR is normal agent work;
never route a mergeable green PR to the owner. A `do-not-automerge` auto-label
(e.g. on workflow-file PRs) holds the enabler, but an agent can still merge such
a PR directly. Leaving finished work parked as a draft is what stranded
#371–#380 — flip ready and land it. For the recreated project the
auto-merge/sweep apparatus is retired entirely; see `docs/NEXT-TASKS.md` →
"Wind-down / retirement".
