## What this repo is

The **websites** repo for the menno420 estate — one repo, multiple Railway
services (share code, not a process):
- **control-plane** (`app/`) — **public** readiness board + journal browser over
  `superbot`, `superbot-next`, `substrate-kit`, and `websites` itself, live from
  the GitHub API. The one non-public datum (Actions secret names) is masked to a
  count on the public board; a password-gated `/owner` area ([D-0012]) un-masks
  it and exposes privileged actions (cache refresh, re-run failed CI). Full doc:
  `docs/site.md`.
- **botsite** (`botsite/`) — the public marketing + reference site. Doc:
  `docs/botsite.md`.
- **dashboard** (`dashboard/`) — the **public** developer dashboard: read-only
  oversight of the bot's own inventory (functions/commands/settings/access/env/
  ideas/bugs), with the live-bot-write control panel stubbed by design. Doc:
  `docs/dashboard.md`.
- **review** (`review/`) — the program-review site built for Anthropic
  reviewers (owner-directed 2026-07-11, expanded same day): process /
  growth / successes / problems / fleet index + per-repo detail /
  continuous review editions (+ Atom feed) / evidence-backed questionnaire
  + questions ledger, rendered server-side from committed data mirrors
  (`review/data/*.json` + `review/data/reviews/*.md`, baked by
  `review/gen_snapshot.py`, `gen_fleet.py`, `gen_stats.py`; refreshed by
  the scheduled `review-bake` workflow). Read-only, network-free at
  runtime. Full doc: `review/README.md`. The Railway service is **LIVE** at
  https://review-production-f027.up.railway.app (owner-created 2026-07-12 —
  `docs/owner/OWNER-ACTIONS.md` Decided row J).

**Bake pipeline reality (2026-07-13):** the scheduled `review-bake`
workflow WORKS end-to-end — first scheduled SUCCESS ever recorded
2026-07-13 (run 29235587736; data PR #259 = `bcf2943`, then #270 =
`5b9c6f7`), and since #269 it also refreshes botsite's
`botsite/data/graveyard.json` and dispatch-chains a quality run onto its
own bake PR. Standing limitation: the bake PR is created with the ambient
`GITHUB_TOKEN`, so it gets NO `pull_request`-event checks (GitHub's
recursion guard) and the ruleset refuses the dispatch-run substitute with
verbatim `405 Repository rule violations found` (measured chain in
`docs/CAPABILITIES.md`, #274). **Interim landing path:** a session
closes/reopens the bake PR (fires a real quality run) and a non-author
review-merges it. **Durable fix:** the `BAKE_PAT` ⚑ ask in
`docs/owner/OWNER-ACTIONS.md` (one fine-grained PAT, two scopes, stored as
an Actions secret; review-bake's `GH_TOKEN` then switches to it).

All FOUR services are **public** — the Basic-auth gate was dropped from
control-plane + dashboard ([D-0011]); botsite and review were always
public. The single gated corner is the control-plane `/owner` area.

Deployment: `docs/deployment.md` + each service's doc.

  `unknown`), and all three services expose an unauthenticated `/version` JSON.

  is read live from `RAILWAY_GIT_COMMIT_SHA` (primary, Railway-injected) →
  `GIT_SHA` (Docker build-arg fallback) → `"unknown"`. No secret / Railway token.

  `python3 bootstrap.py check --strict --require-session-log`, the
  `scripts/check_no_ambient_railway_ids.py` guard, plus every service's pytest
  suite. **`quality` is now a REQUIRED status check on `main`** (owner set the
  ruleset 2026-07-09; verified live on PR #18) — PRs are blocked until it is
  green. The session-gate step is **PR-diff-aware** (since kit v1.0.0, [D-0017]):
  it derives the session card from the PR/push diff and passes it via
  `check --session-log <card>`, so a stale `complete` card can't mask THIS
  PR's `in-progress` one (`fetch-depth: 0` on checkout feeds the diff). The
  vendored `bootstrap.py` is kit **v1.17.0** (stepwise §4.3 upgrades
  v1.6.0 → v1.17.0, PR trail #62…#199 then the v1.16.0/v1.17.0 upgrade

  `bootstrap.py` header) whose checker fails a born-red

  let PR #19 auto-merge on a born-red card alone is closed.
