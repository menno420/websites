## [D-0004] Control-plane data/auth/cache model
- status: decided
- date: 2026-07-09
- verdict: All GitHub data fetched live server-side: REST API for listings/state (GITHUB_TOKEN), raw.githubusercontent.com WITHOUT auth for public file bodies; per-URL in-memory TTL cache 180s caching successes and stable negatives (404/403/401) but never transient errors; ?refresh=1 busts. ## [D-0010] console.json consumed under a pinned, versioned cross-repo shape contract
- status: decided
- date: 2026-07-09
- verdict: The `/console` page's feed (superbot's committed `botsite/data/console.json`) is no longer consumed on an implicit schema. Superbot commits the canonical versioned contract (`botsite/data/console_data_contract.json`, superbot PR #1884: top-level families + guaranteed fields per record + `version`), stamps `meta.schema_version` into every emitted feed, and enforces producer-side parity + fail-closed shape checks in its CI. This repo pins the contract copy it was built against (`dashboard/console_data_contract.json`, v1) and verifies at render time (`data_source.console_contract_issue`): version mismatch or a missing contracted family renders an honest schema-drift banner on `/console` (never-fake-data posture) instead of a silently wrong page. Shape changes flow as: superbot edits the canonical contract + bumps version (explicit, reviewable) → this repo syncs the pinned copy + adapts templates/fixture in one commit.
- why: The two repos shared an implicit schema — a producer-side family/field rename would silently blank or corrupt the consumer page (superbot's BUG-0022 desync class) with no signal on either side. The first pass of writing the consumer against the real contract immediately proved the point: the console page treated `ideas`/`bugs` as lists (`|length`) while the feed ships counter dicts (`{total, by_status, open_count, open}`), so its stat tiles showed dict-key counts — fixed in the same PR (template + route + fixture now match the contracted shape, plus an Open-bugs list the dict shape provides for free). Enforcement is deliberately split: field-level checks live producer-side in superbot's CI (where the shape is constructed); the consumer runs the two cheap render-time checks that make drift visible where the owner looks.
- provenance: websites PR #11 (console-feed-contract session, 2026-07-09); superbot PR #1884 + superbot `.sessions/2026-07-09-kl6-console-telemetry.md` (the idea's origin); docs/dashboard.md § Data source

## [D-0011] Control-plane + dashboard made PUBLIC (Basic auth dropped); readiness secrets cell masked to a count
- status: decided
- date: 2026-07-09
- verdict: The two formerly-private websites services are now **public** — the HTTP Basic auth gate is removed from both `control-plane` (`app/main.py`) and `dashboard` (`dashboard/app.py`). Every route serves without credentials; the `_authorized` helper + `auth_gate` middleware + now-dead `base64`/`secrets` imports are deleted, `/healthz` still works, and the apps no longer fail-closed 503 when `SITE_PASSWORD` is absent (they run fully public). `SITE_PASSWORD` is left set-but-unused in Railway (non-destructive, reversible); the dead `dashboard/config.py` (which only held it) is removed and `app/config.py` drops the var. The **one security refinement**: because the control-plane readiness board renders GitHub Actions **secret NAMES** (obtainable only with an admin-scope token — the single non-public datum on an otherwise public-repo-derived board), the secrets cell now shows a **COUNT only** (`N secret(s)`). The individual names are never rendered in the board HTML nor serialized into `/api/readiness.json` (the raw secrets API response is no longer carried in the returned row). Everything else on the board (rulesets, required checks, CODEOWNERS presence, auto-merge, open PRs, journal contents) is derived from public repos and is unchanged. Tests rewritten for the public model; a new assertion verifies a secret name (`ANTHROPIC_API_KEY`) is absent from the served board HTML and JSON. `botsite` was already public and is untouched. Deployed non-destructively to the `superbot-websites` project only (RAILWAY_API_KEY + explicit superbot-websites IDs; ambient production-bot IDs never passed; no destructive mutation).
- why: The owner directed it in-session, verbatim "Yes drop the auth" — the passwords were judged unnecessary friction, and the surfaces are read-only oversight of public data (superbot's committed JSON artifacts + public repo config), so there is nothing to gate. The single genuinely non-public datum, the Actions secret names, is masked to a count rather than dropped so the board keeps its "N secrets configured?" signal without exposing what they are. Owner-directed work merges immediately on green (Q-0191-class); the change is reversible (re-add the middleware / the var is still set) and covered by tests.
- provenance: websites PR #12 (drop-auth session, 2026-07-09); owner verbatim "Yes drop the auth"; docs/site.md § Auth, docs/dashboard.md, docs/deployment.md
