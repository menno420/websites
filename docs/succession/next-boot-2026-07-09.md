## 3. Walking-skeleton check — prove the pipeline before real work

Before any substantive change, prove the full loop on a trivial change
(a one-line docs touch is enough): **branch → READY PR → CI green →
auto-merge → merged.** Gen-1 lost time to every link in this chain at least
once; a fresh Project must not assume any of them.

The repo's ritual, exactly:

1. `git fetch && git checkout -b claude/<slug> origin/main` — **clones here
   start on a detached HEAD at main's tip; branch before committing.**
2. **First commit = the born-red session card**: `.sessions/<date>-<slug>.md`
   with `> **Status:** \`in-progress\``. This holds CI red **by design** —
   the `quality` gate derives the card from the PR diff and runs
   `python3 bootstrap.py check --strict --require-session-log --session-log <card>`.
3. Push the branch **over git** (an MCP-only PR can sit with zero check runs),
   open the PR **READY**, and arm auto-merge yourself
   (`mcp__github__enable_pr_auto_merge`, SQUASH) — MCP-created PRs don't
   trigger any enabler. Alternative repo convention since PR #31: MCP
   squash-merge after `quality` goes green.
4. Do the work; verify (§5).
5. **Deliberate last step: flip the card.** The gate's hold tokens live in
   `bootstrap.py` (`IN_PROGRESS_TOKENS = ("in-progress", "in progress",
   "wip", "hold", "drafted")`) — a card counts complete only when its Status
   badge carries none of those (**write `complete`**), every required marker
   is present (`📊 Model:` line, `## 💡 Session idea`,
   `## ⟲ Previous-session review`), and no `[[fill:]]` slots remain.
   The exact red message you'll see until then (verbatim, from PR #46):
   `check: session log … is missing: Session idea, Previous-session review,
   a completed Status (badge still says in-progress)`.
6. Confirm the merge with `git ls-remote` — not with MCP reads (§4, stale
   cache). `control/**`-only diffs ride a fast lane (in-job green
   short-circuit, no card needed).

Regression coverage for the gate itself: `tests/test_born_red_session_gate.py`.
