Process

The substrate workflow

Agent sessions cannot talk to each other and remember nothing between runs. The workflow turns those limits into structure: every channel is a committed file, every session leaves a record, and every merge passes one enforced gate. What follows is the actual mechanism, not an idealization — the same files and gates are linked as evidence.

The bus: two files, one writer each

Coordination runs over a two-file protocol in control/ (spec):

control/inbox.md — orders in

The manager lane appends numbered ORDER blocks: priority, the ask, why, and a checkable done-when. Only the manager ever writes this file. CI enforces it: an inbox diff must be pure-append and grammatically a well-formed order.

control/status.md — heartbeat out

Each session overwrites this as its deliberate last step: phase, health, last shipped PR, blockers, orders acked/done, and anything only the owner can do. Only this lane ever writes it. Stale heartbeat = the fleet page treats the lane as dark.

One writer per file means two agents never edit the same file, so there are no merge conflicts by construction. The control-plane site renders every lane's heartbeat on one screen — the owner's single glance over a fleet whose sessions are otherwise invisible.

The landing path — what one unit of work walks through

Every box below is enforced by a gate or proven by a committed artifact; none of it is convention-only.

The gate that makes it real

quality — one required CI check on main — runs the workflow engine's strict checker (doc hygiene, session-card markers, heartbeat grammar), repo-specific safety guards, and every service's full test suite. Its sharpest tooth is the born-red session gate: a PR whose session card still says in-progress fails CI by design, so a red first run on every build PR is the system working — the merge unlocks only when the session flips its card complete with the write-up done. Control-file-only diffs (heartbeats, order appends) ride a fast lane that still validates them.

The gates are themselves under test: the born-red gate has a both-directions regression test, and the fast-lane gates are driven end-to-end against the real checker CLI (tests/test_control_gates.py). The problems page tells the story of the day this gate leaked — and how that was found and fixed.

What a session does when nobody routed it work

The work ladder: inbox orders first, then queued work, then the ideas backlog (each session must capture one deduplicated idea, so the backlog self-feeds), then contained self-initiated improvements, then honest upkeep — truth-sweeping stale docs, verifying live deploys, rescuing other sessions' stranded work. "Idle" is not a state; neither is inventing busywork — an upkeep pass that finds nothing says so in the heartbeat.

Unattended operation

Sessions arm their own wake-up clocks: a 4-hourly scheduled routine (self-armed after the manager verified the capability — ORDER 008) plus a self-scheduled timer chain for continuous mode, which landed 26 verified slices and 3 rescues across one 17-hour unattended run. The reliability caveats are real and documented on the problems page.
