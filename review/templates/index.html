The shape of the program, in one paragraph

A fleet of Claude Projects ("lanes"), one per repo, coordinated by a manager lane — with committed git files as the only channel between them: orders arrive in control/inbox.md, state goes out in control/status.md, and one writer per file makes the whole thing conflict-free. Every working session logs itself in a committed card that CI holds red until the session finishes its write-up. Merging to main is the deploy, so a required quality gate — workflow checker plus every service's test suite — stands in front of every merge. Sessions are amnesiac by nature, so anything worth knowing is written down: verified capabilities and walls (with exact error text), decisions, ideas, retrospectives. The result is a program whose entire operating history is inspectable — which is what this site renders.

How to read this site

  • Process — the substrate workflow: the orders bus, born-red session cards, the landing path a change walks from wake to verified deploy, and a glossary for the jargon.
  • Growth — per-day charts (PR merges, agent sessions, test count) derived from git history, with the milestone timeline under them.
  • Successes — what worked, each with links into the committed evidence.
  • Problems — what failed or nearly failed, what it cost, and what changed. Written with the same specificity as the successes, on purpose.

A note on honesty: this program's own retro doctrine is that a review hiding failures is worthless. This site was built by the same kind of agent session it describes, and its numbers come from a committed snapshot of the real history — see the footer for the exact commit, or fetch /story.json.
